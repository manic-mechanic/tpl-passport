# Cross-Platform Deep Dive: Kotlin Multiplatform (KMP)

> Learning-oriented deep dive. See `native-apps.md` for the actual porting strategy.

---

## The Philosophy: "Shared Logic, Native UI"

KMP's core idea is fundamentally different from Flutter and React Native. Instead of writing UI once and rendering it everywhere, you write **business logic once in Kotlin** and let each platform write its own native UI that consumes it.

| Layer | Where it lives |
|---|---|
| API / networking | Shared Kotlin (`commonMain`) |
| Data models + serialization | Shared Kotlin |
| Business logic, validation | Shared Kotlin |
| State / ViewModel | Shared Kotlin |
| Local database | Shared Kotlin (SQLDelight) |
| UI | Native — Compose on Android, SwiftUI or Compose MP on iOS |
| Platform APIs (camera, NFC, GPS) | Platform-specific via `expect`/`actual` |

**Compose Multiplatform** is the extension that lets you share UI too — but this is optional. Many production KMP apps use fully native SwiftUI on iOS and Compose on Android, sharing only the logic layer. This is the architecture Netflix and Cash App use.

The upside over Flutter/RN: the UI is genuinely native — not drawn by a custom renderer, not bridged. A KMP iOS app with SwiftUI UI is indistinguishable from a pure SwiftUI app to the user, because it IS a SwiftUI app, just with the logic layer written in Kotlin.

### vs. Xamarin / MAUI

| | Xamarin / MAUI | KMP |
|---|---|---|
| Language | C# | Kotlin |
| iOS runtime | CLR on device (historically sluggish) | Compiled to native ARM64 — no managed runtime on iOS |
| UI sharing | MAUI attempts full shared UI via XAML renderers | Logic only by default; Compose MP for optional UI sharing |
| Lock-in | Microsoft ecosystem | JetBrains + open Kotlin ecosystem |

KMP's key technical advantage over Xamarin: Kotlin/Native compiles to actual native ARM64 binaries for iOS. There is no CLR or Mono runtime on device. The generated code is called from Swift as a native framework.

## Project Structure

Generated by the wizard at `kmp.jetbrains.com`:

```
MyApp/
├── shared/                     ← pure shared Kotlin module
│   └── src/
│       ├── commonMain/         ← all shared logic lives here
│       ├── androidMain/        ← Android-specific implementations
│       └── iosMain/            ← iOS-specific implementations
├── composeApp/                 ← Android app + optional Compose MP
│   └── src/
│       ├── androidMain/
│       └── commonMain/         ← shared Compose UI (if using CMP)
├── iosApp/                     ← Xcode project
│   └── iosApp/
│       ├── ContentView.swift   ← SwiftUI root, embeds Compose or calls shared code
│       └── iOSApp.swift
└── gradle/
    └── libs.versions.toml      ← version catalog (modern Gradle dep management)
```

You spend most of your time in `shared/src/commonMain/`. The platform-specific folders are only for `actual` implementations of platform-specific behaviour.

## The Kotlin Language

### Coroutines and Flow — the Vue reactivity equivalent

Kotlin's async model maps closely to both JS async/await and Vue's reactivity system:

| Vue / JS | Kotlin |
|---|---|
| `async/await` | `suspend fun` + `coroutineScope` |
| `Promise<T>` | `Deferred<T>` / `suspend fun` returning T |
| `ref(value)` | `MutableStateFlow(value)` |
| `computed(() => ...)` | `.map { }` on a Flow, or `derivedStateOf` |
| `watch(source, cb)` | `.collect { }` on a Flow |
| `watchEffect` | `.collectLatest { }` |
| Pinia store state | `StateFlow` exposed from ViewModel |

```kotlin
// suspend = async. Call with coroutineScope.launch or viewModelScope.launch
suspend fun fetchBranch(id: String): Branch {
    return httpClient.get("$baseUrl/branches/$id").body()
}

// StateFlow = ref() — holds a current value, observable
class PassportViewModel : ViewModel() {
    private val _checkIns = MutableStateFlow<List<CheckIn>>(emptyList())
    val checkIns: StateFlow<List<CheckIn>> = _checkIns.asStateFlow()

    // Derived state — like Pinia getters
    val visitCount: StateFlow<Int> = checkIns
        .map { list -> list.map { it.branchCode }.toSet().size }
        .stateIn(viewModelScope, SharingStarted.Eagerly, 0)

    fun checkIn(branchCode: String) {
        viewModelScope.launch {
            _checkIns.value = _checkIns.value + CheckIn(branchCode, Clock.System.now())
        }
    }
}
```

### Key Kotlin Features Worth Knowing

**Data classes** — pure value holders, auto-generated `equals`/`hashCode`/`copy`:
```kotlin
data class Branch(val code: String, val name: String, val district: String)

// copy() is immutable update — like JS spread
val updated = branch.copy(name = "Updated Name")
```

**Sealed classes** — exhaustive unions, critical for UI state. Compiler forces you to handle every case:
```kotlin
sealed class UiState<out T> {
    object Loading : UiState<Nothing>()
    data class Success<T>(val data: T) : UiState<T>()
    data class Error(val message: String) : UiState<Nothing>()
}

// In Compose — compiler error if you miss a case
when (val state = uiState) {
    is UiState.Loading  -> LoadingSpinner()
    is UiState.Success  -> BranchList(state.data)
    is UiState.Error    -> ErrorMessage(state.message)
}
```

**Extension functions** — add methods to existing types:
```kotlin
fun String.titleCase() = split(" ").joinToString(" ") { it.replaceFirstChar { c -> c.uppercase() } }

"toronto reference library".titleCase()  // "Toronto Reference Library"
```

**Null safety** — like TypeScript strict mode but enforced everywhere with no escape hatches:
```kotlin
val name: String = "Benny"      // cannot be null
val nickname: String? = null    // explicitly nullable
nickname?.length                // safe call — returns Int?
nickname ?: "Anonymous"         // Elvis operator — same as ?? in Swift/JS
```

## Compose Multiplatform — Sharing UI Too

Compose Multiplatform (CMP) extends Android's Jetpack Compose to iOS, desktop, and web. The same composable functions run on all platforms, rendered via Metal on iOS and Vulkan/OpenGL on Android — not native widgets.

**Platform targets as of 2025/2026:**
- Android: stable (Jetpack Compose, stable since 2021)
- iOS: **stable since CMP 1.6.0 (May 2024)**
- Desktop (JVM): stable
- Web (Wasm): Beta

### Compose's Declarative Model — Direct Parallel to Vue

```kotlin
// Compose
@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }
    Button(onClick = { count++ }) {
        Text("$count")
    }
}
```

```vue
<!-- Vue equivalent -->
<script setup>
const count = ref(0)
</script>
<template>
  <button @click="count++">{{ count }}</button>
</template>
```

Both are declarative and reactive. Compose's recomposition tracks `State` reads automatically — same principle as Vue's dependency tracking. The main difference is that Compose is pure Kotlin with no template language; UI is built with function calls.

```kotlin
// v-if / v-for equivalents — just Kotlin
@Composable
fun BranchList(branches: List<Branch>, visited: Set<String>) {
    Column {
        if (branches.isEmpty()) {
            Text("No branches found")
        } else {
            branches.forEach { branch ->
                BranchRow(branch = branch, isVisited = branch.code in visited)
            }
        }
    }
}
```

### Consuming StateFlow in Compose

```kotlin
@Composable
fun PassportScreen(viewModel: PassportViewModel) {
    // collectAsStateWithLifecycle bridges Flow → Compose State
    val checkIns by viewModel.checkIns.collectAsStateWithLifecycle()
    val visitCount by viewModel.visitCount.collectAsStateWithLifecycle()

    Column {
        Text("Visited: $visitCount branches")
        LazyColumn {
            items(checkIns) { checkIn -> CheckInRow(checkIn) }
        }
    }
}
```

`collectAsStateWithLifecycle()` is the bridge between the reactive logic layer (Flow) and the reactive UI layer (Compose). This is the standard pattern for connecting shared ViewModels to Compose screens.

## Networking: Ktor

Ktor is the standard KMP HTTP client. Same API on all platforms, different underlying engines (OkHttp on Android, Darwin on iOS):

```kotlin
// shared/src/commonMain — runs on all platforms
val httpClient = HttpClient {
    install(ContentNegotiation) { json() }
    install(DefaultRequest) {
        header("apikey", SUPABASE_ANON_KEY)
        header("Authorization", "Bearer $SUPABASE_ANON_KEY")
    }
}

class BranchRepository(private val client: HttpClient) {
    suspend fun getCheckIns(userId: String): List<CheckIn> =
        client.get("$SUPABASE_URL/rest/v1/check_ins") {
            parameter("user_id", "eq.$userId")
        }.body()

    suspend fun saveCheckIn(checkIn: CheckIn): CheckIn =
        client.post("$SUPABASE_URL/rest/v1/check_ins") {
            setBody(checkIn)
        }.body()
}
```

For Supabase specifically, the `supabase-kt` library (JetBrains-maintained) provides a higher-level SDK with auth, realtime, storage, and the full REST client — equivalent to the JS `@supabase/supabase-js` SDK.

## Local Storage: SQLDelight

SQLDelight generates typesafe Kotlin from SQL schema files. The Pinia persistence equivalent — but a real database:

```sql
-- shared/src/commonMain/sqldelight/CheckIn.sq
CREATE TABLE CheckIn (
    id         TEXT    NOT NULL PRIMARY KEY,
    branchCode TEXT    NOT NULL,
    timestamp  INTEGER NOT NULL,
    note       TEXT
);

selectAll:
SELECT * FROM CheckIn ORDER BY timestamp DESC;

selectByBranch:
SELECT * FROM CheckIn WHERE branchCode = :branchCode;

insert:
INSERT INTO CheckIn VALUES (?, ?, ?, ?);
```

SQLDelight generates a `CheckInQueries` class — all queries are typesafe Kotlin functions. Works on Android (SQLite driver), iOS (native SQLite), and web (IndexedDB-backed). This is a significant step up from `localStorage` persistence — actual relational queries, schema migrations, type safety.

## Dependency Injection: Koin

Koin handles the "single shared instance" problem that Pinia solves for stores. For a Vue developer, DI is unfamiliar — think of it as "how you share class instances across your app":

```kotlin
// Declare what can be shared
val appModule = module {
    single    { HttpClient { /* config */ } }
    single    { BranchRepository(get()) }
    viewModel { PassportViewModel(get()) }
}

// In Compose — equivalent to usePassportStore() in Vue
val viewModel: PassportViewModel = koinViewModel()
```

Koin is the recommended DI for KMP — it doesn't require annotation processing so it works cleanly in `commonMain`.

## iOS Interop

### How Kotlin Code Reaches Swift

Building the shared module produces an `XCFramework` — a native iOS framework. Swift imports it:

```swift
import shared  // the generated framework

let repo = BranchRepository()
let branches = try await repo.getBranches()  // with SKIE — see below
```

### The Swift API — Clean For Simple Cases, Awkward For Others

Simple data classes and functions generate clean Swift APIs. The problems emerge with Kotlin generics, sealed classes, and coroutines:

**Sealed classes** generate a class hierarchy in Swift, not Swift enums with associated values — verbose to pattern match.

**Coroutines** don't map to Swift's `async/await` without a wrapper — they require callback-style bridges.

**The fix: SKIE** (Swift Kotlin Interface Enhancer by Touchlab). It post-processes the generated framework to produce proper Swift enums for sealed classes and proper `async/await` for suspend functions:

```swift
// Without SKIE — awkward
if let success = state as? UiStateSuccess<NSArray> {
    let data = success.data as! [Branch]
}

// With SKIE — natural Swift
switch state {
case .loading: showSpinner()
case .success(let data): render(data.branches)
case .error(let msg): showError(msg)
}

// Suspend functions become Swift async/await with SKIE
let branches = try await repository.getBranches()

// Kotlin Flow becomes AsyncSequence with SKIE
for await checkIn in viewModel.checkIns {
    updateUI(checkIn)
}
```

SKIE is open source, actively maintained, and considered essential on any serious KMP iOS project. Treat it as a required dependency.

### What Still Requires Swift

Even with Compose MP handling the UI, the iOS app entry point and platform-specific APIs need Swift:
- `@main` app struct
- Push notification registration / APNs
- Camera / AVFoundation
- NFC (CoreNFC)
- Keychain access
- Biometrics (LocalAuthentication)
- App lifecycle (AppDelegate/SceneDelegate)
- Widgets (WidgetKit)

Your Swift experience is genuinely useful here. Compose MP is embedded as a `UIViewController` via `ComposeUIViewController`, so you're still writing Swift for the host layer.

## The `expect`/`actual` Pattern

For platform-specific implementations behind a common interface:

```kotlin
// commonMain — declares the contract
expect class PlatformStorage() {
    fun getString(key: String): String?
    fun setString(key: String, value: String)
}

// androidMain — Android implementation
actual class PlatformStorage() {
    private val prefs = // SharedPreferences
    actual fun getString(key: String) = prefs.getString(key, null)
    actual fun setString(key: String, value: String) =
        prefs.edit().putString(key, value).apply()
}

// iosMain — iOS implementation
actual class PlatformStorage() {
    actual fun getString(key: String) =
        NSUserDefaults.standardUserDefaults.stringForKey(key)
    actual fun setString(key: String, value: String) =
        NSUserDefaults.standardUserDefaults.setString(value, forKey = key)
}
```

Everything in `commonMain` sees `PlatformStorage` as a single unified type. The correct implementation is linked at compile time per platform.

## Web Target Reality

Compose for Web uses Kotlin/Wasm + Skia rendered to a `<canvas>`. Same tradeoffs as Flutter web:
- Not HTML/CSS — pixels drawn to canvas
- No SEO
- Limited accessibility (no native DOM semantics)
- ~3–5MB Wasm bundle
- Browser support requires WASM GC: Chrome 119+, Firefox 120+, Safari 18 (iOS 18+)
- Beta as of 2025 — the web target is not a first-class citizen

**Verdict for this project:** The web PWA stays Vue/Nuxt. KMP web is not a replacement for a purpose-built web app. The practical KMP story is Android + iOS.

## Tooling

**Gradle** — historically painful. Meaningfully improved since ~2022:
- Version catalogs (`libs.versions.toml`) replaced string-based deps
- Kotlin DSL (`.gradle.kts`) gives full IDE autocomplete
- Configuration cache is stable — rebuilds are faster

Still more verbose than npm. Error messages when Kotlin/Native targets break can be cryptic. The `kmp.jetbrains.com` wizard generates a working setup — start there.

**IDE:** Android Studio for Android-primary projects (best Compose preview tooling). IntelliJ IDEA for non-Android targets. Xcode is still required for iOS builds — unavoidable.

**Build speed:** Kotlin/Native compilation for iOS is the biggest practical pain point. Cold builds on a complex shared module can take 2–5 minutes. Incremental builds are fast, but the first build and after any native change is slow. Being actively improved by JetBrains.

## Real Talk

### Is KMP Production-Ready?

**Yes for Android + iOS logic sharing.** Verified production users:
- **Netflix** — shared networking and business logic
- **Cash App** (Block) — SQLDelight and Molecule are their own open-source tools from this work
- **McDonald's** — global app
- **Philips** — medical device companion apps
- **Multiple JetBrains products**

Compose Multiplatform iOS reached "stable" in mid-2024 — not just marketing. JetBrains uses it in their own products.

### Friction Points Tutorials Don't Warn About

1. **iOS build pipeline setup.** Getting the XCFramework into Xcode (SPM vs CocoaPods, path config, Xcode version pinning) is mechanical but error-prone. You will spend time here.
2. **Kotlin/Native compilation speed.** iOS cold builds are slow. Real productivity tax during development.
3. **Debugging across the boundary.** Kotlin breakpoints in iOS simulator work but are less polished than native Swift debugging in Xcode.
4. **Third-party library gaps.** When you need an SDK that only has an Android/iOS native version (some Firebase services, some analytics tools), you write `expect`/`actual` wrappers yourself or find a community wrapper of unknown maintenance status.
5. **Camera / NFC require native code.** There's no built-in KMP abstraction. You write `expect`/`actual` wrapping AVFoundation (iOS) and CameraX (Android). More work than Capacitor's plugin model.

### How KMP Benefits Someone with Both Kotlin and Swift Experience

The core benefit: write networking, models, and business logic once. A bug fix in the shared layer propagates to both platforms simultaneously. "Is this bug in the Android version or iOS version?" becomes a less common question.

For a solo developer working across both platforms, the productivity gain is significant — you're not mentally context-switching between two implementations of the same business rules. The platforms still diverge at the UI layer, but that divergence is the *intended* design.

## Vue → KMP Mental Model Map

```
Pinia store          →  ViewModel + StateFlow
ref(value)           →  MutableStateFlow(value)
computed(() => ...)  →  .map { }.stateIn(scope, ...)
watch(source, cb)    →  .collect { }
$fetch / useFetch    →  Ktor HttpClient (suspend fun)
localStorage persist →  SQLDelight
provide/inject       →  Koin dependency injection
v-if                 →  if (cond) Composable() else Other()
v-for                →  items.forEach { Composable(it) } or LazyColumn
<style scoped>       →  Modifier chain + MaterialTheme tokens
CSS custom props     →  MaterialTheme.colorScheme.primary etc.
```
